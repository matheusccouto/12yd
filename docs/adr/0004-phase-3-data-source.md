# Phase 3 data source: club Shootout Kicks via FotMob

The v3 model loses to a uniform-random dive policy on the LOTO CV aggregate (0.374 vs 0.405, within one aggregate SE of 0.036 — see `docs/model-card-v3.md` § "Cross-validation" and `docs/model-review.md` Topic 5). The aggregate SE scales as `1/sqrt(n)`; the 179 training rows have too little statistical power to distinguish the model from random. The scraper (Phase 2) has closed the URL-rotation wall and documented the 6 untouchable empty-shotmap cases (Issue #49, `data/empty_shotmap_documentation.md`); the v3 training set is now stable at 36 reachable in-scope shootouts × ~5 kicks = 179 rows. **Phase 3 is the path to statistical power.** We chose **club Shootout Kicks via FotMob** (Copa Libertadores, UCL knockout, domestic cup finals) as the Phase 3 data source over a non-FotMob source (StatsBomb Open Data, RSSSF detail pages) because the schema is identical (the same `ShootoutKick` extraction runs on the same `pageProps.content.shotmap.shots` payload), the client is unchanged, the validator is unchanged, and the new training rows roughly double the dataset (~180 expected) without introducing a new client or a new shape. The 6 empty-shotmap cases (Burkina Faso vs Gabon, Mali vs Equatorial Guinea, Cameroon vs Egypt, Burkina Faso vs Cameroon in AFCON 2021; Tajikistan vs UAE, Saudi Arabia vs South Korea in Asian Cup 2023) stay documented as FotMob data gaps; closing them requires leaving FotMob and is deferred to a future Phase 4+ ADR.

## Rationale — the three candidate sources

**1. Club shootouts via FotMob (CHOSEN).** Copa Libertadores (FotMob league 41, slug `copa-libertadores`), UCL knockout (league 42, slug `champions-league`), and the major domestic cup finals — FA Cup (132, `fa-cup`), Coupe de France (133, `coupe-de-france`), DFB-Pokal (125, `dfb-pokal`), Coppa Italia (137, `coppa-italia`), Copa del Rey (138, `copa-del-rey`). Pro: the schema is identical to the in-scope international Shootout Kicks (the `pageProps.content.shotmap.shots` array has the same shape, the `penaltyShootoutEvents` join works the same, the `is_on_target` flag is identical), so the existing `extract_shootout_kicks` runs unchanged. The existing `FotMobClient`, the existing ETag/gzip cache, the existing `FetchResult` envelope, and the existing `validate_shootout_count` all carry over. The same 6 RSSSF-style verifiability applies (RSSSF publishes Copa Libertadores + UCL + domestic cup shootouts in the same per-tournament format as international shootouts). Con: club Shootout Kicks are rarer than international shootouts (the in-game stakes are the same — a knockout shootout — but clubs play fewer knockout matches per season than national teams play in a cup cycle). The expected training-row yield is ~180 rows over 6 seasons (2021–2026) across 5–6 tournaments; that brings the dataset from 179 to ~360, dropping the LOTO CV aggregate SE from 0.036 to ~0.025 (a 30% reduction). The in-game stakes differ (club career vs national-team career), so a `tournament_kind` attribute ∈ {`international`, `club}` is added to the training row so the LOTO CV can group or slice by kind.

**2. StatsBomb Open Data (rejected for Phase 3; deferred).** StatsBomb publishes 360-frame event data including penalty kicks with shooter, keeper, outcome, and the goal-mouth coordinates. Pro: would close the 6 empty-shotmap FotMob gaps entirely (StatsBomb covers AFCON 2021 + Asian Cup 2023 international shootouts), and the coordinate schema is finer than FotMob's 3-class bucketing — it would let us regress on the continuous `x` coordinate in addition to the discrete Side. Con: schema divergence (StatsBomb uses a 360-frame coordinate system, a different per-row join key, and a different match-id space; the extraction path is a new client, a new validator, and a new league registry), a new ingest script, and a new ADR per future tournament covered. The coordinate divergence is the deal-breaker for Phase 3: the v3 model is a multiclass classifier on the 3-class Side bucketing (`side(x)`); joining StatsBomb would require a separate continuous-`x` model alongside the discrete one, and the deployment policy (`argmin(P(L), P(C), P(R))`) doesn't generalise to a continuous-`x` prediction. The right way to do StatsBomb is a separate Phase 4 workstream (a continuous-`x` regression head, a per-keeper dive-direction feature, a 360-frame coordinate bucketing helper), not a Phase 3 bolt-on.

**3. RSSSF detail pages (rejected for Phase 3; deferred).** RSSSF publishes per-shootout line-by-line kicker lists with the kicker, the score, and the outcome, but not the goal-mouth coordinate — i.e. RSSSF gives you *who kicked* and *whether they scored*, but not *where they aimed*. Pro: covers all 6 empty-shotmap cases and the 18 URL-rotation failures (a custom HTML scraper would replace the broken FotMob URL lookup), and the data is verifier-true (the RSSSF oracle is the ground truth for shootout count). Con: no goal-mouth coordinate means every row is `x=NaN` and the L/C/R label is unobservable, so the row cannot join the training table. RSSSF is a *verification oracle* and a *kicker identity* source, not a *Side label* source — adding it to the training table requires the goal-mouth coordinate from somewhere else (StatsBomb, or a third-party dataset like Instat/Opta). The Phase 3 decision is about more *training rows*; RSSSF alone cannot supply them.

## Schema — what changes

The `ShootoutKick` and `TrainingRow` types are unchanged in their existing fields. The only new field is `tournament_kind: Literal["international", "club"]` on `TrainingRow` (default `"international"` for the existing rows, derived from the league registry at extraction time). The `tournament_kind` is propagated to the `predictions.jsonl` artifact only as a per-kicker metadata field (not as a model input) so the dashboard can surface the kind on the per-kicker card in a future iteration. The 17 model features (A1 rolling side counts, A2 last_side, A3 preferred_foot, A4 career_penalty_count, B1 kick_number, B2 score+is_decisive, C1 position) carry over unchanged. No new client, no new feature, no new categorical level.

The `LEAGUE_SEASONS_PREDICT_WINDOW` constant in `src/twelveyards/tournaments.py` is extended with the 5–6 new club (league_id, season) pairs; the per-tournament tests in `tests/test_tournaments.py` add a parallel `CLUB_EXPECTED_SHOOTOUT_COUNTS` map (the per-(league, season) RSSSF count) and a per-pair `CLUB_EXCLUDED_EMPTY_SHOTMAP` map (mirroring the existing `EXCLUDED_EMPTY_SHOTMAP` for international tournaments, in case any club tournaments have empty FotMob shotmaps — the diagnostic file `data/skipped_refs_diagnostics.jsonl` is reused as-is). `RSSSF_TO_LEAGUE_NAME` adds the 5–6 new heading strings (Copa Libertadores, UEFA Champions League, etc.) to the existing 6-heading international map.

The new JSONL rows (`data/shootout_kicks.jsonl`) carry the same shape as the existing rows; the on-disk format is unchanged. The only consumer-visible change is the new `tournament_kind` field on the `TrainingRow` (and the corresponding column in the training table if a CSV export is added later — not in Phase 3).

## LOTO CV grouping — what changes

The existing LOTO CV folds on `tournament_name` (6 folds, one per in-scope tournament; see `docs/model-card-v3.md` § "Cross-validation"). Phase 3 keeps the per-`tournament_name` fold unit — the new club tournaments become additional folds (5–6 new folds × 5–15 kicks each = ~30–90 new holdout kicks), and the aggregate SE drops from 0.036 to ~0.025. The `tournament_kind` attribute is a per-row analysis axis, not a fold unit: a future iteration can ask "what's the per-`tournament_kind` save rate?" or "leave one kind out and train on the other", but those are not the Phase 3 default. The Phase 3 default is "fold on tournament name across all 11–12 tournaments", which is the simplest extension of the existing CV and avoids introducing a `tournament_kind`-based stratification that the model would have to be re-fitted to support.

The cross-tournament aggregate is the v3 headline (`docs/model-card-v3.md` § "Cross-validation"): the model is "0.031 below" the closed-form random baseline (0.374 vs 0.405) on n=179 aggregate, with the SE putting the difference within noise. Phase 3's close condition (Issue #51) is **LOTO CV aggregate SE ≤ 0.025** (a 30% reduction from the v3 0.036), which the ~360-row dataset is expected to satisfy. A per-`tournament_kind` breakdown is a Phase 4 question — Phase 3 keeps the CV fold unit on `tournament_name` and treats `tournament_kind` as a metadata attribute.

## Per-tournament handling — what changes in the leagues registry

The new club tournaments register in `src/twelveyards/leagues.py::EXTENDED_LEAGUES` (the existing player-history-fetcher table; the shootout scraper iterates `LEAGUES`, not `EXTENDED_LEAGUES`). The new `League` records are:

- `League(41, "copa-libertadores", "Copa Libertadores")`
- `League(42, "champions-league", "Champions League")` (the player-history fetcher already has this; we promote it to the shootout scraper's `LEAGUES` list and add the slug)
- `League(132, "fa-cup", "FA Cup")`
- `League(133, "coupe-de-france", "Coupe de France")`
- `League(125, "dfb-pokal", "DFB-Pokal")`
- `League(137, "coppa-italia", "Coppa Italia")`
- `League(138, "copa-del-rey", "Copa del Rey")` (the player-history fetcher already has this)

`LEAGUE_SEASONS_PREDICT_WINDOW` extends with the 6-seasons-per-tournament (2021–2026) pair set. `RSSSF_TO_LEAGUE_NAME` adds the matching heading strings. The `WC_2026_LEAGUE` and `WC_2026_SEASON` constants are unchanged.

## Consequences

- **The training row count roughly doubles.** From 179 rows to ~360 rows across 11–12 tournaments (the 6 international + 5–6 club). The LOTO CV aggregate SE drops from 0.036 to ~0.025 (Issue #51's close condition). The single 28-row WC 2026 holdout is unchanged.
- **The model artifact and the metrics are now produced by a single training run that consumes the combined dataset.** The v3 artifact-vs-metrics data leak (Issue #40) is preserved as a hard constraint: the `lightgbm.pkl` and `model/metrics.json` are fit on the same fold split. The close condition for Phase 3 (Issue #51) requires the LOTO CV aggregate SE to satisfy ≤ 0.025, which is a property of the combined fit, not a property of any single holdout.
- **The 6 empty-shotmap cases (4 in AFCON 2021, 2 in Asian Cup 2023) remain documented gaps.** They are excluded from the expected count (36 reachable in-scope shootouts; the per-pair `EXPECTED_SHOOTOUT_COUNTS` map in `tests/test_tournaments.py` already excludes them). A future Phase 4 ADR can address them via StatsBomb or RSSSF detail scraping.
- **The `tournament_kind` attribute is metadata, not a model input.** The v3 17-feature schema is unchanged. A future iteration could add `tournament_kind` as a categorical input (14 levels vs 11 currently), but the Phase 3 evidence does not motivate the complexity: the international and club shootout distributions are likely similar (both are 3-class side predictions on a per-kicker basis), and the LOTO CV per-`tournament_name` fold already detects per-tournament drift. If the per-`tournament_kind` analysis reveals a real distributional shift, a future ADR can promote it to a model input.
- **The schema divergence with non-FotMob sources is preserved as a future Phase 4 lever.** StatsBomb Open Data is a separate ADR; the 6 empty-shotmap cases can be backfilled then. The Phase 3 path keeps the scraper on a single data shape, which is the path of least resistance to ~360 training rows.
- **HF push remains manual.** A successful re-run + `hf upload couto/12yd . couto/12yd` is the close condition for Issue #51. The model card on HF updates with the new `n_train` / `n_holdout` and the new LOTO CV aggregate SE.
- **The PR for Issue #51 must update the model card on HF** (`docs/model-card-v3.md` → the next versioned card) to lead with the new aggregate SE and the new `n_train`, and to document the new `tournament_kind` attribute in the schema. The frame pin (Kicker-PoV) is unchanged.
